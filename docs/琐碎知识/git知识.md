## 以前已经学过一些基础的了 这里是放实际遇到的问题
1. 推送到远程： 
    ~~~text
   先创建远程仓库 复制ssh  
   然后 推送的时候定义远程仓库
    ~~~
2. 定义远程仓库时 的名称origin
    ~~~text
   因为可以本地 1 对 多个远程 所以得给他们起名字以区分
   默认origin
   ~~~
3. 不要远程仓库了：
    ~~~text
   先控制台 取消关联
   git remote remove origin
   
   然后删除github 库就行
    ~~~
4. 本地修正提交 再推；看来确实是不能实现 不增加提交记录了
    ~~~text
   正常的思维：就是推到远程仓库的流程
   0. 本地完成修改
   1. 拉取远程仓库的最新  (拉取 不知道为什么自动给我 拼一起 写了合并记录)
   2. 看有没有与本地的冲突；没有就推；有就得合并 取舍（或者变基）
   ~~~
   
5. 关于推送 做个总结吧
 ~~~text
   
   这些操作归根结底就是为了 代码不冲突（原本是1版  A改成2  B改成3  AB是同时改的 ；A先一步提交推送了，B此时就得先拉取最新版，解决冲突的地方，再推送）
   而解决冲突 有两种: 合并 变基；
   合并（merge）： 合并总是将 其他分支的修改 合并到 当前所在分支； 即：加上一个新的版本
   变基（rebase）：变基 是指 把当前分支这个修改 变到 其他分支的 最新一次；

   注意处于的情况： 我在A 我是把B合并到我A的最新一次；； 我是A 我是把自己变基到B的最新一次
   补充：一般来说 变基完成后 处于 master为倒数第二个提交，此时切换到master，合并分支，就可以删除分支标签，只要master
    
   下面就理解远程的了：
   1.第一个方式：
     pull：拉取 合并；；即：把远程的合到我的本地所处分支，自然会有新记录（可以解决4中的1了）；
      然后再推送 我的本地分支的 新的合并后的版本；（合并解决的冲突：左：当前分支 中：要产生的合并版 右：远程分支）
      
      拉取并合并
      git pull                     # 相当于 git fetch拉取不自动合并 + git merge合并
   
    2. 第二种：
     我把master 变基到远程分支，解决冲突，然后直接推送；
     （这里的冲突是： 左：当前分支  中：共同祖先  右：远程分支）
      （注意一个问题 就是  比如  fun变基到master后，会把两个分支共祖（4版）后的所有fun变到master后！！！
        4版-》5版  master
           -》4.1版-》4.2版-》4.3版 fun
    所以：我个人的项目：
    1. 合并：master改好了 然后切换到远程分支 合并master；切换到master分支 推送，远程标签就会同步到master最新版
    2. 变基：master改好了 把master变基到远程分支 推送，远程标签就会同步到master最新版
    两种都行；几个人协作的那种 没试过到时候再说
   
   ~~~
> 这是合并：完成C4以后，切换到master分支 将C4合并到master；或者说叫 合并expriment：git merge expriment
